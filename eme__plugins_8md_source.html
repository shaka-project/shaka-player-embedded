1 # Custom EME Plugins
2 
3 This document describes how to create and register custom EME plugins. Defining
4 a custom EME implementation allows the app to handle decrypting the media
5 content without handling media-specific features. As of now, the EME
6 implementations cannot handle decoding; they can only decrypt the bytes. This
7 means you can't implement secure decrypt-decode or secure rendering.
8 
9 
10 ## Overview
11 
12 First, you need to create the custom implementation types. You need to define
13 derived types for [Implementation](@ref shaka::eme::Implementation) and
14 [ImplementationFactory](@ref shaka::eme::ImplementationFactory). There will be
15 a single, global instance of the factory that is used to create instance of the
16 Implementation. Each playback instance will get its own Implementation instance
17 and there can be multiple alive at once if there are multiple Player instances.
18 
19 ### Registering your plugin
20 
21 Once you have written the type, you need to compile it into some library. You
22 have two options:
23 
24 First, you can compile your code into your app. You can do so any way you like.
25 Then, before you create a Player instance, you need to create a global instance
26 of your ImplementationFactory and register it with the
27 [ImplementationRegistry](@ref shaka::eme::ImplementationRegistry). This factory
28 instance must live for the duration of the app.
29 
30 Second, you can compile your code into a separate static library. Then you use
31 the plugin definition below to register with our build system. When you build
32 Shaka Player Embedded, our build system will include your library and include
33 the plugin into the Embedded library. The implementation(s) will be
34 automatically registered during library startup.
35

37 ## Plugin Definition
38 
39 If you want, you can create a JSON definition of the plugin and have it built
40 into the library. This means the plugin will be auto-registered and you just
41 have to use the library like normal. This also allows you to keep the library
42 and the plugin together and avoid versioning problems.
43 
44 To define a plugin, you write a JSON file that describes your plugin. It should
45 contain one or more static libraries that will be built into the library. Then
46 you indicate the headers where the implementations are found and we will
47 generate code to register those types.
48 
49 To register your JSON file, pass the `--eme-impl` flag to `./configure`. You
50 can pass this multiple times for multiple implementations:
51 
52 ```sh
53 ./configure --eme-impl my-eme-def.json
54 ```

101 #### Example
102 
103 ```json
104 {
105  "libraries": [
106  "build/cdm_adapter.a"
107  ],
108  "implementations": [
109  {
110  "factory_type": "my_ns::MyImplementationFactory",
111  "header": "include/my_implementation_factory.h",
112  "key_system": "com.example.cdm"
113  }
114  ]
115 }
116 ```

118 #### Factory requirements
119 
120 Each implementation factory defined by the plugin must publicly implement the
121 ImplementationFactory type and must have an empty constructor. The factory will
122 be allocated using a global static variable. Since the code is being linked
123 statically, the type doesn't need to be exported.
124 
125 #### Paths
126 
127 File paths in the JSON file can be absolute; if they are relative, they are
128 relative to the JSON file. File paths can include format specifiers to give
129 different paths based on what is currently being built. The format specifier is
130 in the format `%(name)s` where name is one of the below names. For example,
131 `foo/%(config)s/bar` may give `foo/Debug/bar`.
132 
133 | Name | Description | Possible values |
134 | ---- | ----------- | --------------- |
135 | config | The current configuration name | `Debug` or `Release` |
136 | arch | The target CPU architecture | `x86`, `x64`, `arm`, or `arm64` |
137 | os | The target OS | `linux`, `mac`, `win`, `ios`, or `android` |
138 
139 #### Building plugin from source
140 
141 It is possible to build your plugin from source instead of from a static
142 library. If you pass the `sources` field, those files will be compiled into
143 the library. This allows access to the library internals. For example, this
144 can be used so your plugin can use the
145 [FileSystem](@ref shaka::util::FileSystem) type to avoid duplicating code. Note
146 that internals are not ABI or API stable, so they may change at any time.
